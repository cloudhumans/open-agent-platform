# Stream Metadata

When the Open Agent Platform frontend invokes your agent via `stream.submit()`, it sends a `metadata` object alongside the messages and config. Your agent can read this metadata at runtime to make context-aware decisions.

## Metadata Fields

Every stream invocation includes the following metadata:

| Field | Type | Always Sent | Description |
|-------|------|-------------|-------------|
| `supabaseAccessToken` | `string \| undefined` | Yes | The authenticated user's access token (from Cognito/Supabase session). Use this to make authenticated calls to downstream services on behalf of the user. |
| `tenant_id` | `string \| undefined` | No | The selected tenant identifier. Only sent when the agent is the **Agent Creator** (`agentId === NEXT_PUBLIC_AGENT_CREATOR_ID`). |
| `project` | `string \| undefined` | No | The selected project within the tenant. Only sent when the agent is the **Agent Creator**. Passed via query string from the project selection modal. |

## When Each Field Is Sent

### Always (all agents)

```typescript
metadata: {
  supabaseAccessToken: "eyJhbGciOiJSUzI1NiIs..."
}
```

### Agent Creator mode only

When the user clicks **"Get Agentic Done"** from the agents page, selects a project, and starts a chat, the frontend adds tenant and project context:

```typescript
metadata: {
  supabaseAccessToken: "eyJhbGciOiJSUzI1NiIs...",
  tenant_id: "printi",
  project: "printicomercial"
}
```

## Data Flow

The Agent Creator flow works as follows:

1. **User clicks "Get Agentic Done"** on the agents page header
2. **Project selection modal** shows the tenant's available projects (from `claudiaProjectIds`)
3. **Frontend navigates** to `/?agentId=<AGENT_CREATOR_ID>&deploymentId=<AGENT_CREATOR_DEPLOYMENT_ID>&project=<selected_project>`
4. **Thread component** reads `agentId`, `deploymentId`, and `project` from query params
5. **On every `stream.submit()`**, `tenant_id` and `project` are included in the metadata object

### Example Tenant Data

To help understand the values your agent will receive, here are example tenants:

```json
[
  {
    "tenantName": "printi",
    "claudiaProjects": [
      "printi",
      "printicomercial",
      "printisuporte",
      "evento_scrapper_printi_produtos",
      "printicomercialdemo"
    ]
  },
  {
    "tenantName": "get-agentic-done",
    "claudiaProjects": ["gad"]
  }
]
```

In this example:
- `tenant_id` would be `"printi"` or `"get-agentic-done"`
- `project` would be one of the values from `claudiaProjects`, e.g. `"printicomercial"`

## Accessing Metadata in Your Agent

### Python (LangGraph)

```python
from langchain_core.runnables import RunnableConfig

def my_node(state: State, config: RunnableConfig):
    metadata = config.get("metadata", {})
    tenant_id = metadata.get("tenant_id")
    project = metadata.get("project")
    access_token = metadata.get("supabaseAccessToken")

    # Use tenant/project context for your agent logic
    if tenant_id and project:
        # Agent Creator mode - you know which tenant/project to operate on
        pass
```

### TypeScript (LangGraph)

```typescript
import { RunnableConfig } from "@langchain/core/runnables";

function myNode(state: State, config: RunnableConfig) {
  const metadata = config.metadata ?? {};

  const project = metadata.project;
  const accessToken = metadata.supabaseAccessToken;

}
```

## Agent Creator UI Behavior

When the frontend detects that the current agent is the Agent Creator, it also adjusts the UI:

- **Tool calls are hidden** (`hideToolCalls` is forced to `true`)
- **Textarea is pre-populated** with a default message
- **New thread**, **file upload**, and **hide tool calls toggle** controls are hidden
- The chat operates in a simplified, conversational mode focused on agent creation
